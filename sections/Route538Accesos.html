8. Gestión de Seguridad y Accesos

8.1 Objetivo

La Gestión de Seguridad y Accesos tiene como objetivo garantizar la confidencialidad, integridad y disponibilidad del servicio DNS, asegurando que los accesos estén correctamente definidos, controlados y auditables, conforme a las políticas de seguridad de la organización y a las mejores prácticas de AWS.

8.2 Alcance

Esta sección aplica a:

  • Accesos al servicio DNS en Amazon Route 53.
  • Definición de roles y permisos por función organizacional.
  • Gestión de identidades y accesos.
  • Auditoría y control del uso de privilegios.

8.3 Principios de seguridad

  • Mínimo privilegio: cada rol cuenta únicamente con los permisos necesarios.
  • Separación de funciones: ningún rol concentra funciones críticas incompatibles.
  • Trazabilidad: todas las acciones deben ser auditables.
  • Acceso basado en roles: no se otorgan permisos directos a usuarios.
  • Gobierno centralizado: la definición de accesos sigue un modelo controlado.

8.4 Modelo de control de accesos en AWS (Route 53)

El control de acceso al servicio DNS se realiza mediante AWS IAM, utilizando roles y políticas asociadas a funciones organizacionales.

Principios clave:

  • No se utilizan usuarios individuales para la operación del servicio.
  • El acceso se otorga mediante roles IAM.
  • Los roles son asumidos de forma temporal.
  • Todas las acciones quedan registradas en los servicios de auditoría de AWS.
  • La definición final de nombres y estructuras de permisos es responsabilidad de Public Cloud (AWS).

8.5 Definición de perfiles de acceso

A continuación se describe el perfil funcional de cada grupo y los permisos asociados a Route 53. Los permisos listados representan capacidades funcionales, no implementaciones técnicas finales.

8.5.1 Public Cloud (AWS)

Rol: Administración de plataforma y control de accesos.

Responsabilidades:

  • Definir la estructura de roles IAM.
  • Validar la nomenclatura y consistencia de los permisos.
  • Administrar el acceso a Route 53 a nivel plataforma.

Permisos funcionales típicos:

  • route53:CreateHostedZone
  • route53:DeleteHostedZone
  • route53:AssociateVPCWithHostedZone
  • route53:DisassociateVPCFromHostedZone
  • route53:ListHostedZones
  • route53:GetHostedZone

8.5.2 Network Platform – Network Services

Rol: Gobierno técnico del servicio DNS.

Responsabilidades:

  • Definir cómo debe operar el servicio DNS.
  • Validar cambios estructurales.
  • Ejecutar tareas operativas en caso de escalamiento desde IT Ops.

Permisos funcionales típicos:

  • route53:ListHostedZones
  • route53:GetHostedZone
  • route53:ListResourceRecordSets
  • route53:ChangeResourceRecordSets
  • route53:GetChange

8.5.3 IT Ops – Network Services

Rol: Operación diaria del servicio DNS.

Responsabilidades:

  • Ejecución de cambios operativos.
  • Atención de solicitudes e incidentes.
  • Validación básica del servicio.

Permisos funcionales típicos:

  • route53:ChangeResourceRecordSets
  • route53:GetChange
  • route53:ListResourceRecordSets

8.5.4 Network Platform – NetFactory

Rol: Automatización y despliegue controlado.

Responsabilidades:

  • Ejecución de automatizaciones.
  • Integración con pipelines CI/CD.
  • Uso de roles técnicos de corta duración.

Permisos funcionales típicos:

  • route53:ChangeResourceRecordSets
  • route53:GetChange

8.5.5 CISO

Rol: Gobierno, riesgo y cumplimiento.

Responsabilidades:

  • Supervisión de controles de acceso.
  • Aprobación de excepciones.
  • Revisión de cumplimiento normativo.

Permisos funcionales típicos:

  • route53:ListHostedZones
  • route53:GetHostedZone
  • Acceso a registros y auditoría (solo lectura).

8.6 Gestión de identidades y accesos

  • Los accesos deben integrarse con el sistema corporativo de identidad.
  • Se debe aplicar MFA cuando esté disponible.
  • Los accesos deben revisarse periódicamente.
  • Las bajas y cambios deben aplicarse de forma inmediata.

8.7 Monitoreo y auditoría de accesos

  • Todas las acciones deben quedar registradas.
  • Los eventos deben integrarse con los procesos de auditoría.
  • Accesos sospechosos deben tratarse como incidentes de seguridad.

8.8 Gestión de excepciones

  • Las excepciones deben documentarse y justificarse.
  • Requieren aprobación del CISO.
  • Deben tener fecha de expiración y revisión periódica.
