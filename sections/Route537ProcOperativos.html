7. Procesos Operativos

7.1 Gestión de Cambios

7.1.1 Objetivo

El proceso de Gestión de Cambios tiene como objetivo asegurar que todas las modificaciones realizadas sobre el servicio DNS se ejecuten de forma controlada, trazable y segura, minimizando el riesgo de interrupciones, errores operativos o impactos negativos al negocio.

7.1.2 Alcance

Este proceso aplica a todos los cambios relacionados con:

  • Registros DNS.
  • Políticas de enrutamiento.
  • Automatizaciones asociadas al servicio.
  • Configuraciones que impacten el comportamiento del DNS.

Todos los cambios deberán ser gestionados y registrados a través de la herramienta corporativa de ITSM: ServiceNow.

7.1.3 Clasificación de cambios

Cambios estándar

  • Cambios repetitivos, de bajo riesgo y previamente aprobados.
  • Actualización de registros DNS existentes.
  • Cambios operativos sin impacto arquitectónico.

Responsabilidades:

  • Responsible: IT Ops – Network Services
  • Accountable: Network Platform – Network Services

Cambios complejos / de arquitectura

  • Modificación de políticas de enrutamiento.
  • Cambios estructurales en la arquitectura DNS.
  • Introducción de nuevos patrones operativos.

Responsabilidades:

  • Responsible: Network Platform – Network Services
  • Accountable: CISO

Cambios automatizados

  • Cambios ejecutados mediante flujos de automatización previamente aprobados.

Subcategorías:

  • Diseño y desarrollo: Network Platform – NetFactory (Responsible) / Network Platform – Network Services (Accountable)
  • Ejecución operativa: IT Ops – Network Services (Responsible) / Network Platform – Network Services (Accountable)

7.1.4 Flujo del proceso de cambio

  1. Registro del cambio en ServiceNow.
  2. Clasificación del tipo de cambio.
  3. Evaluación de impacto técnico y de seguridad.
  4. Aprobación conforme al tipo de cambio.
  5. Ejecución del cambio.
  6. Validación post-cambio.
  7. Cierre y documentación en ServiceNow.

7.1.5 Controles y trazabilidad

  • Registro en ServiceNow.
  • Evidencia de aprobación.
  • Evidencia de ejecución y validación.
  • Auditoría y trazabilidad completa.

7.1.6 Gestión de riesgos asociados a cambios

  • Evaluación previa de riesgos.
  • Aprobación del CISO para cambios de alto impacto.
  • Definición de planes de reversión.
  • Documentación del impacto al negocio.

7.1.7 Roles involucrados

RolParticipación
Public Cloud (AWS)Consultado cuando el cambio impacta la plataforma
IT Ops – Network ServicesEjecución y validación operativa
Network Platform – Network ServicesGobierno técnico y aprobación
Network Platform – NetFactoryAutomatización y soporte técnico
CISOAprobación de cambios de alto riesgo

7.2 Gestión de Incidentes

7.2.1 Objetivo

El proceso de Gestión de Incidentes tiene como objetivo restablecer la operación normal del servicio DNS lo antes posible, minimizando el impacto al negocio y asegurando una comunicación clara, trazable y coordinada entre los equipos involucrados.

Este proceso está orientado a la restauración del servicio, no a la identificación de causa raíz.

7.2.2 Alcance

  • Fallas en la resolución de nombres.
  • Impacto en aplicaciones derivado de problemas DNS.
  • Errores de configuración.
  • Fallas relacionadas con automatizaciones.
  • Incidentes derivados de cambios recientes.

7.2.3 Clasificación de incidentes

Incidente menor

  • Impacto limitado o localizado.
  • No afecta servicios críticos.
  • Resolución mediante procedimientos estándar.

Incidente mayor

  • Impacta servicios críticos o múltiples usuarios.
  • Requiere coordinación entre equipos.
  • Puede involucrar a Public Cloud o CISO.

7.2.4 Flujo de gestión de incidentes

  1. Detección y registro.
  2. Clasificación y priorización.
  3. Diagnóstico inicial.
  4. Escalamiento.
  5. Resolución y recuperación.
  6. Cierre documentado.

7.2.5 Roles y responsabilidades

RolResponsabilidad
IT Ops – Network ServicesAtención inicial y resolución
Network Platform – Network ServicesSoporte especializado
Public Cloud (AWS)Soporte cuando impacta la plataforma
Network Platform – NetFactoryApoyo técnico
CISOGestión de incidentes de seguridad

7.2.6 Comunicación y escalamiento

  • Comunicación centralizada vía ServiceNow.
  • Escalamiento según criticidad.
  • Documentación obligatoria.

7.2.7 Seguimiento y mejora

  • Análisis de causas recurrentes.
  • Definición de acciones preventivas.
  • Revisión periódica.

7.3 Gestión de Solicitudes

7.3.1 Objetivo

Atender solicitudes relacionadas con el servicio DNS de forma controlada, trazable y eficiente, sin representar un incidente ni un cambio complejo.

7.3.2 Alcance

  • Creación o modificación de registros DNS estándar.
  • Solicitudes de información.
  • Validaciones operativas básicas.

7.3.3 Tipos de solicitudes

Solicitudes estándar

  • Altas, bajas o modificaciones simples.
  • Consultas sobre configuraciones.
  • Verificación de resolución DNS.

Solicitudes especiales

  • Solicitudes con impacto potencial.
  • Requieren validación técnica previa.

7.3.4 Flujo de atención

  1. Registro en ServiceNow.
  2. Clasificación y validación.
  3. Asignación al equipo correspondiente.
  4. Ejecución.
  5. Validación y cierre.

7.3.5 Roles y responsabilidades

RolResponsabilidad
IT Ops – Network ServicesEjecución de solicitudes estándar
Network Platform – Network ServicesValidación técnica
Network Platform – NetFactorySoporte de automatización
Public Cloud (AWS)Consulta cuando impacta la plataforma
CISOConsulta en temas de seguridad

7.3.6 Controles y trazabilidad

  • Registro obligatorio en ServiceNow.
  • Evidencia de aprobación.
  • Auditoría y trazabilidad completa.

7.4 Gestión de Riesgos Operativos

7.4.1 Objetivo

Identificar, evaluar, mitigar y monitorear los riesgos asociados a la operación del servicio DNS.

7.4.2 Alcance

  • Operación diaria del servicio DNS.
  • Cambios operativos y técnicos.
  • Automatizaciones y despliegues.
  • Incidentes y eventos recurrentes.

7.4.3 Tipos de riesgos considerados

Riesgos operativos

  • Errores humanos.
  • Configuraciones incorrectas.
  • Fallas en ejecución de cambios.

Riesgos técnicos

  • Fallas en resolución DNS.
  • Errores derivados de automatizaciones.
  • Degradación del servicio.

Riesgos de seguridad

  • Accesos no autorizados.
  • Cambios no aprobados.
  • Exposición de información sensible.

7.4.4 Gestión del riesgo

  1. Identificación del riesgo.
  2. Registro en el sistema Heracles.
  3. Evaluación del impacto.
  4. Definición de acciones de mitigación.
  5. Aprobación del riesgo.
  6. Ejecución y seguimiento.

7.4.5 Roles y responsabilidades

RolResponsabilidad
IT Ops – Network ServicesIdentificación temprana de riesgos
Network Platform – Network ServicesEvaluación técnica y mitigación
Public Cloud (AWS)Mitigación de riesgos de plataforma
Network Platform – NetFactoryImplementación de acciones
CISOAprobación y gobierno de riesgos

7.4.6 Integración con otros procesos

  • Gestión de Cambios
  • Gestión de Incidentes
  • Gestión de Seguridad
  • Auditoría

7.4.7 Registro y seguimiento

  • Todos los riesgos deben documentarse en Heracles.
  • Cada riesgo debe contar con responsable y plan de tratamiento.
  • Revisión periódica y actualización continua.
