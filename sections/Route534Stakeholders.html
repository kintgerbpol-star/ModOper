4. Stakeholders

4.1 Objetivo de la sección

Definir los actores involucrados en la provisión, operación, gobierno y seguridad del servicio de DNS, así como sus responsabilidades y límites de actuación, con el fin de asegurar una operación controlada, trazable y alineada con los objetivos del negocio.

4.2 Actores involucrados

El servicio de DNS involucra a los siguientes equipos y roles:

  • Public Cloud (AWS)
  • Network Services
  • IT Ops – Network Services
  • Network Platform – Network Services
  • Network Platform – NetFactory
  • CISO / Seguridad de la Información
  • Áreas consumidoras del servicio (aplicaciones, plataformas, etc.)

4.3 Descripción de roles y responsabilidades

4.3.1 Public Cloud (AWS)

Rol: Proveedor y responsable de la plataforma DNS.

Responsabilidades principales:

  • Creación y eliminación de Hosted Zones.
  • Asociación de Hosted Zones a VPCs.
  • Definición y aplicación de políticas de enrutamiento.
  • Administración de la plataforma AWS subyacente.
  • Garantizar la disponibilidad del servicio DNS desde la perspectiva del proveedor.
  • Proveer visibilidad, auditoría y controles de acceso (IAM, CloudTrail, etc.).

Fuera de alcance:

  • Operación diaria de registros DNS.
  • Gestión de incidentes operativos.
  • Definición de arquitectura funcional del servicio.

4.3.2 IT Ops – Network Services

Rol: Operación diaria del servicio DNS.

Responsabilidades principales:

  • Gestión operativa de registros DNS.
  • Atención y resolución de incidentes relacionados con DNS.
  • Ejecución de cambios estándar aprobados.
  • Validación post-cambio y monitoreo básico.
  • Escalamiento oportuno a Network Platform o Public Cloud cuando aplique.

Fuera de alcance:

  • Diseño de arquitectura DNS.
  • Implementación de automatizaciones.
  • Definición de políticas de enrutamiento.

4.3.3 Network Platform – Network Services

Rol: Gobierno técnico del servicio DNS.

Responsabilidades principales:

  • Ejecución de la arquitectura DNS.
  • Definición de estándares técnicos y mejores prácticas.
  • Aprobación técnica de cambios complejos.
  • Soporte de segundo nivel.
  • Validación de impacto técnico y escalabilidad.

Fuera de alcance:

  • Operación diaria del servicio.
  • Ejecución directa de cambios operativos.

4.3.4 Network Platform – NetFactory

Rol: Automatización y habilitación técnica.

Responsabilidades principales:

  • Desarrollo de automatizaciones e infraestructura como código (IaC).
  • Integración de DNS con pipelines CI/CD.
  • Despliegue controlado de cambios automatizados.
  • Mantenimiento de herramientas de automatización.

Fuera de alcance:

  • Operación diaria del servicio.
  • Definición de arquitectura DNS.
  • Gestión directa de incidentes.

4.3.5 CISO / Seguridad de la Información

Rol: Gobierno de seguridad y gestión de riesgos.

Responsabilidades principales:

  • Definición de políticas de seguridad aplicables al servicio DNS.
  • Aprobación de cambios de alto impacto o riesgo.
  • Supervisión del cumplimiento normativo.
  • Gestión de riesgos y excepciones.

Fuera de alcance:

  • Operación técnica del servicio.
  • Implementación directa de cambios.

4.4 Relación entre los roles

El modelo se basa en una separación clara de responsabilidades:

  • Public Cloud (AWS) provee la plataforma.
  • Network Platform – Network Services define cómo debe operar el servicio.
  • IT Ops – Network Services ejecuta la operación diaria.
  • Network Platform – NetFactory automatiza y estandariza.
  • CISO gobierna y valida el riesgo.

Este enfoque permite escalabilidad, control y continuidad operativa.

4.5 Principios de colaboración

  • Todas las actividades deben ser trazables y auditables.
  • Ningún equipo opera fuera de su ámbito definido.
  • Los cambios deben seguir procesos formales.
  • La comunicación entre equipos es obligatoria ante eventos de impacto.
